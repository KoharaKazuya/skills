# KoharaKazuya/skills

Claude Code、GitHub Copilot、Codex、Cursor、Antigravity などのコーディング AI に再利用可能な rules / skills を与えるためのリポジトリです。

このリポジトリは、単なるプロンプト集ではありません。
PdM やプロダクトオーナーが、コーディング AI を通じて直接扱えるソフトウェア生成システムを構築するための運用知識をまとめます。

## 背景

エンジニアリングのうち、プログラミングに近い部分は、次第に次の問いへ近づいています。

**PdM が直接触れるソフトウェア生成システムを、いかに構築するか。**

このシステムは、Claude Code などのコーディング AI を中心としたツール群によって実現されます。

コーディング AI は、コードを書く道具にとどまりません。
プロダクトの意図を読み、コードベースを調査し、テストを実行し、実装を変更し、リスクを報告する実行主体です。

そのため、コードだけでは不十分です。
コーディング AI が適切に動作するには、プロダクト意図、意思決定、保証すべき振る舞い、検証方法が、エージェントから参照できる形で存在している必要があります。

## 目的

このリポジトリの目的は、コーディング AI を安全かつ継続的に使うための再利用可能な rules / skills を提供することです。

主な対象は以下です。

* エージェントに常時適用する方針
* 特定タスクで呼び出す作業手順
* 実装前に検証方法を確立するための作業規約
* コード化できない情報をドキュメントとして扱うための方針
* エージェントが自律的にテストするための方針

## 基本方針

### コード化できない情報をドキュメントとして保持する

コーディング AI は、参照できない情報を正しく扱えません。

コード化できない情報は、エージェントが読めるドキュメントとしてリポジトリ内に保持します。
基本的には Markdown を使用します。

最低限、プロダクトリポジトリには次の資料を置きます。

* ユーザーストーリー
* ワイヤーフレーム
* ADR
* 自動テスト

これらは、人間だけのための補助資料ではありません。
エージェントが実装判断を行うための入力です。

### 重要な意思決定を ADR として記録する

重要な意思決定は ADR として記録します。

最低限、ADR には次の3項目を含めます。

1. **Context**: この意思決定が必要になった背景・目的・制約・前提を記載する。
2. **Decision**: 採用する方針・方式・ルールを、後から判定できる形で記載する。
3. **Alternatives**: 検討した選択肢と、それぞれを選ぶ／選ばないことで生じる利点・欠点・リスクを記載する。

ADR の目的は、文書量を増やすことではありません。
将来の実装判断を追跡可能にすることです。

### 保証すべき振る舞いを自動テストにする

保証すべき振る舞いは、自動テストとして表現します。

手動確認は補助的な手段です。
主な保証は、自動化された検証で行います。

自動テストは、品質保証だけでなく、プロダクト仕様の一部です。

### 実装前に検証方法を確立する

コードを変更する前に、エージェントは意図した振る舞いをどう検証するかを明確にします。

原則として、次の順序で作業します。

1. 保証すべき振る舞いを特定する。
2. 最小の検証方法を決める。
3. 必要に応じて自動テストを追加または更新する。
4. 最小限のコード変更を行う。
5. 関連する検証を実行する。
6. 検証できたこと、検証できなかったこと、残るリスクを報告する。

検証方法が不明なまま実装を始めるべきではありません。

### 手動確認より自動検証を優先する

エージェントは、手動確認よりも自動テスト、スクリプト、再現可能なコマンドによる検証を優先します。

手動確認が必要な場合は、その理由を明示します。

望ましい状態は、エージェントが自律的に次を実行できることです。

* 変更する
* 検証する
* 結果を報告する
* 残るリスクを明示する

### オーナーのエンジニアが説明できる状態を維持する

コーディング AI が多くの実装を担う場合でも、オーナーのエンジニアは主要な仕組みを説明できる必要があります。

最低限、説明できるべきものは以下です。

* 主要なアーキテクチャー
* データフロー
* テスト方針
* 重要なトレードオフ
* 失敗しやすい箇所
* エージェントが誤った前提を置きやすい箇所

エージェントだけが理解し、エンジニアが説明できないコードベースは望ましくありません。

## インストール

### skills

Agent Skill のインストーラーが利用できる場合は、それを使います。

```sh
gh skill preview KoharaKazuya/skills <skill-name>
gh skill install KoharaKazuya/skills <skill-name> --agent claude-code --scope user
```

### rules

rules は Agent Skill ではありません。
常時適用する方針です。

各ツール向けの adapter をコピーして使います。

Claude Code:

```sh
cp adapters/claude-code/CLAUDE.md /path/to/your/project/CLAUDE.md
```

Codex:

```sh
cp adapters/codex/AGENTS.md /path/to/your/project/AGENTS.md
```

GitHub Copilot:

```sh
mkdir -p /path/to/your/project/.github
cp adapters/github-copilot/.github/copilot-instructions.md /path/to/your/project/.github/copilot-instructions.md
```

既存の指示ファイルがある場合は、上書きせず差分を確認してください。

## 利用方針

このリポジトリは、コーディング AI 利用の共通基盤として扱います。

プロジェクト固有の指示は、このリポジトリではなく対象プロジェクト側に置きます。

このリポジトリに含めるべきもの:

* 汎用的な検証方針
* 汎用的なテスト方針
* Git 操作の安全方針
* 不具合修正の共通手順
* TDD 実装の共通手順
* PR レビューの共通観点
* コードベース理解を支援する手順

このリポジトリに含めるべきでないもの:

* 会社内部の情報
* 特定プロダクトのビジネスルール
* 非公開リポジトリ名
* 認証情報
* 顧客情報
* 一度きりの実装計画
* 一時的なタスクメモ

## 目指す状態

このリポジトリが前提とする理想状態は、次のとおりです。

* プロダクト意図がドキュメント化されている。
* 重要な意思決定が ADR として記録されている。
* 保証すべき振る舞いが自動テストで検証されている。
* エージェントが必要な文脈をリポジトリ内から参照できる。
* エージェントが自律的に実装と検証を実行できる。
* オーナーのエンジニアが主要な仕組みを説明できる。

これは、コーディング AI を単なるコード生成ツールではなく、ソフトウェア生成システムとして扱うための基盤です。
